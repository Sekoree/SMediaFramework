## Phase P0 — Critical correctness & stability

- [ ] **P0-1 FFmpeg demux stop race + interrupt callback.** **Deferred (needs a focused native pass).** Requires installing an `AVIOInterruptCB` on `_fmt` driven by `_demuxerStopRequest` (so `av_read_frame` aborts promptly and the 4 s join can't time out into a free/seek-while-blocked race), plus not clearing `_demuxerThread`/`_demuxerStopRequest` until the thread exits. Not done this round: hand-writing the native function-pointer interop against FFmpeg.AutoGen 8's binding without source to verify the callback type, and without a slow/hanging-I/O harness to test it, is too risky to ship blind. Partial mitigation already in place: P0-5b's demux fault boundary means a faulting/aborted read no longer crashes the host.
- [x] **P0-2 VideoPlayer post-submit double-dispose (3 sites).** **Closed via** capturing the PTS before `Submit`, gating counters/notification on a `submitted` flag, and routing the event through a new `RaisePresented` helper that wraps the invocation in its own try/catch (never disposes the frame, never lets a subscriber exception reach the clock-driver thread) — applied in `OnVideoTick`, `PresentLatestQueuedFrame`, `TrySubmitHeldFrame`. Note: `VideoFrame.Dispose` is idempotent (`Interlocked.Exchange`), so the real symptom was a *premature* release of a frame an async output still held queued, not a literal double-free. Regression test `VideoPlayerTests.Submit_Success_With_Throwing_PresentationEvent_Does_Not_Release_Frame`. **Verification correction (2026-06-02):** the double-release was fixed in all three sites, but `PresentLatestQueuedFrame` (the `LatestOnTick` path, called from `OnVideoTick` on the clock-driver thread) was still doing a *direct* `FramePresentationTimePresented?.Invoke` rather than going through `RaisePresented` — so a throwing subscriber could still escape to the clock driver there. Now routed through `RaisePresented` like the other two sites.
- [x] **P0-3 VideoRouter.RemoveOutput leaks owned outputs.** **Closed via** capturing the registration under `_gate`, then disposing it (when `DisposeOutputOnRouterDispose`) *outside* the lock — `VideoOutputPump.Dispose` joins its drainer thread, so holding `_gate` during teardown is unsafe.
- [x] **P0-4 AudioRouter drains unrouted/unstarted sources.** **Closed via** gating the run-loop read on routing: only sources targeted by a route this chunk are read (reused per-chunk `HashSet` so a source feeding many routes is read once; muted-but-routed sources still advance — mixer mute semantics). `keepRunning` only auto-stops once routes exist AND every routed source is exhausted, so dynamic "route-last" graphs aren't killed and an empty/route-less router idles instead of stopping. MediaPlayer now routes its audio source to a `DiscardingAudioOutput` so "play to completion" / master-clock advance still works with no real output attached (a later clocked PortAudio output still becomes the pacing primary). Regression test `AudioRouterSourceGatingTests.RunLoop_UnroutedSource_IsNotConsumedUntilRouted`.
- [x] **P0-5a AudioRouter background rethrow → fault.** **Closed via** a `Faulted` event + `Fault` property (new `AudioRouterFaultedEventArgs`); `RunLoop`'s catch now records the fault and raises the event instead of rethrowing (the `finally` still tears down thread/cts); `Start` clears the prior fault. Regression test `AudioRouterFaultTests.RunLoop_SourceThrows_RouterFaultsInsteadOfCrashing`.
- [x] **P0-5b FFmpeg demux thread exception boundary.** **Closed via** wrapping `DemuxerThreadProc` in a catch that records `_demuxFault` (exposed as `DemuxFault`), signals EOF (`_fileReadCompleted = true`) and pulses `_queueGate` so blocked audio/video tracks wake and exhaust gracefully; `StartDemuxerThread` clears the fault.
- [x] **P0-5c NDI receiver fault boundaries.** **Closed via** a `_faultEx` field + `Fault` property + `Faulted` event on both receivers; `IsExhausted` now terminal on fault; `CaptureLoop` wrapped in try/catch (video uses a `CaptureLoopCore` split to avoid re-indenting) that records the fault, wakes the blocked video reader, and stops the router pulling audio.
- [x] **P0-6 Static/text/image frame lifetime.** **Closed.** `TextLayerSource` fixed via a refcounted generation buffer (`FrameBuffer : IDisposable`): each rasterise produces an immutable pooled buffer; emitted frames share it and hold a ref (no per-frame copy while static); a property change rasterises a new generation, and an old one returns to the pool only when the source and all referencing frames are disposed. Regression tests `TextLayerSourceTests.HeldFrame_NotMutated_ByLaterTextChange` / `HeldFrame_RemainsReadable_AfterSourceDispose`. **Remainder closed 2026-06-02:** `StaticFrameSource` and `ImageFileSource` now refcount their shared backing too (source ref + one per live frame; the release hook / `ArrayPool.Return` fires only once the source AND all emitted frames are disposed). This became a *real* use-after-free once P2-14 made `StaticFrameSource.FromFrame` always pass `owned.Dispose` (an `ArrayPool`-backed clone) as the release hook, and `ImageFileSource` always pools — disposing either mid-playback returned a buffer a queued frame still aliased. `StaticFrameSource` only refcounts when a release hook is present, so the plain managed-array path stays zero-overhead (GC-safe). Regression tests `StaticFrameSourceTests.Dispose_WithReleaseHook_DefersReleaseUntilInFlightFramesDisposed` / `ReleaseHook_FiresOnce_WhenFrameOutlivesSource_AndFrameDisposeIsIdempotent`.

## Phase P1 — Transport, sync, decode/encode

- [ ] **P1-1 Coordinated seek as default.** **Deferred (UX/architecture decision).** The coordinator's `SeekCoordinated` *pauses and stays paused*, so making it the default `MediaPlayer.Seek` would turn seek-during-playback into stop-at-position (a UX change). The correct fix — for shared demux, `SeekPresentation` once then reset both tracks/clocks, and pause→seek→resume-if-was-playing — lives in the session layer and should be designed deliberately, not a one-line default swap. The race half is already addressed by P1-3.
- [x] **P1-2 Pause order.** **Closed via** pausing audio (router + clock, or the video clock when audio-less) first, then doing the possibly-blocking `video.Pause` in a `finally`, with the shared-mux flush still last.
- [x] **P1-3 Shared-demux seek audio lock.** **Closed via** wrapping the audio decode-state flush/reset in `SeekPresentation` (`_aCtx` flush, `swr_close`/`swr_init`, counters) in `lock (_audioDecodeLock)` — the same lock `AudioTrack.ReadInto` holds. The video consume below takes `_videoDecodeLock` sequentially, so no nested-lock-ordering hazard.
- [x] **P1-4 Reset fallback PTS counters on seek.** **Closed via** re-anchoring `_vFramesEmitted` (shared demux `SeekPresentation`) and `_framesEmitted` (`VideoFileDecoder.Seek`) to `round(position * fps)` so no-PTS streams resume at ~seek-target instead of a stale future index.
- [x] **P1-5 AudioClipVoice EOF release hang.** **Closed via** finishing the release ramp immediately (`_gain = 0; _stopped = true`) when the source is exhausted mid-release, so `IsExhausted` becomes true instead of lingering. Regression test `AudioClipTests.Voice_StopAtEndOfClip_BecomesExhausted`.
- [x] **P1-6 FFmpeg video encoder allocate-once.** **Closed via** allocating `_frame`'s plane buffers once in `OpenCodecLocked` (format/dims set there), checking the `av_frame_make_writable` return in `Submit`, and removing the per-frame `av_frame_get_buffer` from `FfmpegAvFrameFill.CopyVideoFrame` (now a pure copy with a dimension guard).
- [x] **P1-7 FFmpeg audio encoder non-FLT/FLTP.** **Closed via** a per-encoder `SwrContext` (FLT-in → codec-sample-fmt-out) used in `WriteFrameLocked`, replacing the FLTP-only / raw-float-copy branches. Now correct for FLAC (s16/s32) and any integer/planar codec. Regression test `FFmpegMuxRoundTripTests.Audio_flac_round_trip_preserves_signal` (lossless RMS check).
- [x] **P1-8 Router clock catch-up cap.** **Closed via** re-anchoring the deadline to "now" once the backlog exceeds `MaxCatchupChunks` (64, matching `MediaClock`) in both `WallClockRouterClock` and `PlaybackSlavedRouterClock`.
- [ ] **P1-9 Frame-mode audio tail drain.** **Deferred — no current consumers.** `IAudioSource.TryReadNextFrame(out AudioFrame)` is a default interface method returning false; nothing in the framework or tests consumes audio in frame mode (the router uses `ReadInto`, which *does* drain the swr tail correctly). Fixing dead code now is low value; revisit if/when a frame-mode audio consumer (e.g. a future encode pull path) lands.
- [x] **P1-10 ResamplingAudioSource seekability.** **Closed via** a `ResamplingAudioSource.Create` factory that returns a `SeekableResamplingAudioSource : ResamplingAudioSource, ISeekableSource` when the inner is seekable (forwards Seek/Position/Duration and flushes the resampler on seek via `ResetAfterInnerSeek`); the FFmpeg auto-resample factory now uses `Create`. Regression tests in `ResamplingAudioSourceTests` (incl. `Router_AutoResample_PreservesSeekThroughWrapper`).
- [~] **P1-11 NDI ingest timing.** **Verified, deferred with rationale (2026-06-02).** The finding is literally true — `NDIVideoReceiver.CaptureLoopCore` derives video PTS from a synthetic fixed cadence (`_nextPts += _ptsStep`, `EnsureFormat` sets `_ptsStep` from `format.FrameRate`) and ignores the captured `NDIVideoFrameV2.Timecode`/`Timestamp`. **But the synthetic monotonic PTS is load-bearing, not an oversight:** `RebaseToLatest(playClock.CurrentPosition)` (called in production at `UI/HaPlay/Playback/HaPlayPlaybackSession.cs:1446,1461`) resets `_nextPts` so the continuously-running live receiver presents in sync with the *playback* clock at the moment the operator hits Play. The **audio** ingest clock (`NDIIngestPlaybackClock`, the A/V master) already *does* use NDI timecode/timestamp via `NDIFrameTiming`, so video rides a real master either way. Switching video PTS to NDI absolute time would have to be reconciled with the rebase mechanism (the safe shape: capture the first post-rebase frame's NDI time as a session origin and present `rebaseBase + (frameNdiTime − origin)`, falling back to synthetic when timecode is `TimecodeSynthesize` and timestamp is `TimestampUndefined`). That is a real VFR/dropped-frame correctness improvement but it reshapes a carefully tuned live path and **cannot be hardware-validated in this environment**, so it is deferred deliberately rather than shipped blind. Building blocks already exist and are unit-tested (`NDIFrameTiming.TryMapPresentationTime`).
- [x] **P1-12 VideoPlayer restart guard.** **Closed via** a `_decodeThreadStuck` flag set when `StopInternal`'s bounded join leaves the decode thread `IsAlive` (and it wasn't an early cooperative cancel); `Play()` then throws instead of starting a second decode thread over the same source/queue.

## Phase P2 — Allocation, contracts, API hardening

- [x] **P2-1 Lazy-start output-pump threads.** **Closed via** deferring the pump's drainer `Thread.Start()` (the OS-thread/stack allocation — the cheap `new Thread(...)` managed object stays in the ctor) behind a new idempotent `OutputPump.EnsureStarted()`, guarded against a late-start-after-dispose race by a `_startGate` lock (a thread started after `_ready`/`_cts` are disposed would crash). The router calls `EnsureStarted()` for every registered pump in `Start()` (before the run loop) and for a single pump in `AddOutput` when already running. A router that's constructed + has outputs added but is never started now spawns **zero** pump OS threads — directly removing the suite-level thread pressure behind the intermittent `OutputPump..ctor` OOM. `Dispose`/`WaitForIdle`/`AbandonQueue` are already safe for an unstarted thread (`JoinThread`/`IsAlive` guard on `IsAlive`, false until started). Regression tests `AudioRouterPumpLifecycleTests.OutputAddedBeforeStart_ReceivesAudio_AfterStart` / `OutputAddedWhileRunning_ReceivesAudio` (both fail if either `EnsureStarted` call site is dropped — audio never drains).
- [ ] **P2-2 VideoRouter conversion outside lock.** Snapshot routes/outputs under `_gate`, then convert/readback/submit outside it (`VideoRouter.cs:547-687`, `746-755`). **Verified open (2026-06-02), left deliberately:** `SubmitLocked` does run CPU readback/convert/clone under `_gate`. But (a) the mitigation already noted holds — default outputs are `VideoOutputPump`-wrapped so per-output `Submit` under the lock is a fast enqueue, leaving only branch-output conversion under the lock (a route-change *latency* issue, not a correctness bug); and (b) a safe snapshot-then-convert-outside-lock refactor of the framework's central video data path needs multi-output integration coverage (SDL/GL/NDI) that **can't run in this environment**. Higher regression risk than reward for a headless pass — recommend doing it alongside a real multi-output harness.
- [x] **P2-3 VideoOutputPump reconfigure drain.** **Closed via** dropping queued frames in `Configure` when the new format differs from the inner's current format (the unchanged-format branch-route re-Configure stays a cheap pass-through). Residual: a frame already in-flight on the drainer can still race — noted; full format-versioned queues left as a follow-up.
- [x] **P2-4 VideoOutputPump dispose vs blocked Submit.** **Closed via** checking `Thread.IsAlive` after the join; if the drainer is still blocked in a slow inner `Submit`, `Dispose` now leaks `_pending`/`_cts`/the inner deliberately (logs) instead of disposing them under the running thread (which would throw `ObjectDisposedException` on the drainer or use-after-dispose the inner).
- [x] **P2-5 Compositor hot-path allocations.** **Closed via** serializing the single-consumer read with a `_readGate` and reusing three per-instance scratch lists (slot snapshot, `CompositorLayer` layers, acquired-slot list) across composites, plus **removing the per-slot `SlotFrameLease` object entirely** — `Slot.AcquireLatest`/`AcquireMasterAligned` now return the `VideoFrame?` directly and register a read-ref, and `TryReadNextFrame` tracks the acquired slots and calls a new `Slot.ReleaseFrame()` in its `finally`. A steady-state composite now allocates only the compositor's output frame (slot-count-independent) instead of a `Slot[]` + two `List<>`s + N leases. Regression test `VideoCompositorSourceTests.SteadyStateCompositeRead_AllocationDoesNotScaleWithSlotCount` (measures per-read bytes at 1 vs 8 slots; fails if per-slot allocation returns). **Left as-is:** the public `Slots` getter still allocates a snapshot array per access — it has no hot-path callers (diagnostics only) and the immutable-snapshot return is the safe contract.
- [ ] **P2-6 FFmpeg planar pin/alloc per frame.** `stackalloc`/`fixed`/pooled pinned buffers in the swscale paths (`VideoFileDecoder.cs:745-829`; shared demux `~1280-1363`). **Re-verified 2026-06-02, deferred pending profiling (per the finding's own "measure first").** Confirmed `BuildConvertedPlanarFrame` allocates per converted frame, **but the dominant allocations (`strides` int[], `memories` ReadOnlyMemory[], `buffers` byte[][]) necessarily escape into the returned `VideoFrame`** — they *are* the frame's planes/strides and must live until it's disposed, so they can't be `stackalloc`'d. The only non-escaping scratch is the transient `List<GCHandle>` (freed before return; the GCHandle pinning is brief). A safe reduction is feasible — pool the escaping arrays per-frame and return them on frame dispose, exactly as the pass-through path already does with `PassThroughDescriptorArena` — but it touches the decode hot path for an **unmeasured** gain, so it should follow a profiler showing this path is hot, not precede it.
- [~] **P2-7 FFmpeg pass-through per-plane memory-manager alloc.** **Re-verified 2026-06-02 — largely already mitigated.** Both decoders (`VideoFileDecoder.BuildPassThroughFrame` and `MediaContainerSharedDemux.BuildPassThroughVideoFrame`) already pool the per-frame plane/stride **descriptor arrays** via `PassThroughDescriptorArena` (per-plane-count Treiber free-list, `Array.Clear` on return) — the bulk of the original concern. The only residual per-frame allocation is the small `new UnmanagedMemoryManager<byte>(...)` wrapper per plane (n≤4). Pooling those too requires extending the arena handle to carry + reset the wrappers across the frame's lifetime (the wrapper's `.Memory` is handed into the frame, so it can't be reused until the frame is disposed). Low value, profiling-gated — left for if/when a profiler flags pass-through wrapper churn.
- [x] **P2-8 FFmpeg audio encoder O(n) pending shift.** **Closed via** replacing the `List<float>` + `RemoveRange(0,…)` with a `float[]` + read/write offsets (`AppendPending` compacts/grows only when the tail is full); draining an encoded frame is now O(1). Validated by the P1-7 FLAC round-trip (chunked Submit).
- [ ] **P2-9 CPU compositor straight-alpha.** Encode alpha mode in metadata/pixel format or normalize at source boundaries (`CpuVideoCompositor.cs:16-22`, `172-204`). **Verified open (2026-06-02), deferred:** confirmed the blend treats all BGRA32 as premultiplied. But `VideoFrameMetadata` has no alpha-mode field, and — as the finding's own write-up concedes — every in-framework BGRA producer is already premultiplied or opaque-alpha (SkiaSharp `Premul`, FFmpeg swscale `0xFF`), so the proper fix (add `VideoAlphaMode` to the metadata bundle + a straight→premultiplied normalize in the blend) adds public API surface that **no current producer would set**. Low value until a real straight-alpha source exists; revisit then.

## Phase P3 — Architecture (design-level; scope per item)

- [x] **P3-1 Owned MediaSession/PlaybackSession.** **Closed via** a new `MediaSession` (`S.Media.Playback`) — a single owning handle around a `MediaPlayer` plus the resources wired around it. `MediaSession.Owning(player)` disposes the player; `Borrowing(player)` leaves it to the caller. `Own<T>(resource)` registers extra outputs / companion hosts / app disposables (the Borrow escape hatch is simply *not* registering). Disposal is ordered: the player first (so its routers stop before anything still feeding an output is freed — building on P2-18's player-owned companions), then registered resources in reverse order. Implements `IAsyncDisposable` (which `MediaPlayer` alone does not) so a resource needing a native drain (encoder/file output, router flush) can be awaited; `Dispose` stays for synchronous hosts. Added `MediaPlayer.IsDisposed`. Regression tests `MediaSessionTests` ×4 (player-first-then-reverse order, Borrow leaves the player, `DisposeAsync` awaits `IAsyncDisposable`, idempotent + `Own`-after-dispose throws). Note: this is additive — the existing builder ownership flags (`WithVideoLead(disposeOnPlayerDispose)`, `WithPortAudio(transferHostOwnershipToPlayer)`) remain; `MediaSession` is the opt-in umbrella for apps that want one object to dispose.
- [x] **P3-2 First-class clips/voices/cues.** **Closed via** a new `Soundboard` (`S.Media.Core.Audio`) over the `AudioRouter`: `AddCue(id, clip, output, mode, chokeGroup, gain, …)` registers a named cue (backed by the existing `AudioClipPlayer` for mode/choke/poly/reap), and `Fire(id)` returns a `CueVoice` handle — `Stop()` (release fade), `SetGain()` (router route gain, click-free), `Position`, `IsActive`, and a `Completed` event raised once when the voice finishes (detected on `Reap`/next `Fire`). `StopAll` / `StopGroup` / `Reap`, plus ordered `Dispose` (hard-stops voices so a borrowed router's sources/choke regs are removed immediately; disposes the router when `ownsRouter`). Added `AudioClipPlayer.TryFire` (yields the voice + source id; `Fire` now delegates to it). **Latent bug fixed along the way:** `AudioClip.CreateVoice(null)` / `AudioClipPlayer.Fire(options: null)` produced **silent** voices because `default(AudioClipVoiceOptions)` zero-inits `StartGain` to 0 (the ctor's `= 1f` only applies via the primary ctor) — added `AudioClipVoiceOptions.Default` (full gain) and used it at both `?? default` sites. Regression tests `SoundboardTests` ×4 (audio delivered + `Completed`, choke-stops-previous, unknown→null, dispose hard-stops + disposes owned router). **MVP scope note:** `FadeTo(duration)` and per-voice `Seek` from the original wish-list are not included — the router's gain ramp is click-free but not duration-parameterized, and `AudioClipVoice` has no seek; both are additive follow-ups. Voice *fault* events are N/A (clip voices read from an in-memory buffer and can't fault); `Completed` covers lifecycle.
- [x] **P3-3 Per-session options vs process-wide statics.** **Closed via** per-instance / per-call overrides for the genuinely session-scoped policy knobs, statics kept as escape hatch: (1) `AudioRouter.AutoResampleDefault` (nullable instance property) — `AddSource` resolves per-call arg → instance default → static `DefaultAutoResample`, so a session sets its own policy without mutating the static; (2) `VideoCompositorOptions.AutoBackends` — per-call backend factories tried before the process-wide `RegisterAutoBackend` registry. **Deliberately NOT made per-session: `MediaFrameworkPlugins`.** On inspection those slots are *capability registration* (is FFmpeg / SkiaSharp installed in this process), populated once at startup by `.UseFFmpeg()` / `.UseSkiaSharpImages()` and read framework-wide — not session config. There's no realistic case where two sessions in one process want different decoders, and a per-session registry would thread a context object through dozens of read sites for ~zero production value; the documented save/restore-in-`finally` pattern already covers test isolation (see the new test). Regression test `AudioRouterAutoResampleDefaultTests.AutoResampleDefault_Instance_OverridesStatic_AndIsOverriddenByPerCall`.
- [x] **P3-4 Composition time-driven by default.** **Closed via** making the declarative `VideoCompositor` advance layers to a master timeline instead of one frame per downstream read. **Two modes:** with a `Clock` set (multi-layer programs / transitions) it's *timeline-driven* — each `LayerHandle` keeps a **bounded per-layer look-ahead queue** (the finding's "decode layers into bounded PTS queues") and on each composite catches the layer up to the clock position (bounded pull cap), selects the frame whose interval contains the master time (frame-accurate), and re-submits to the slot only when that frame changes (a downstream player prebuffering its queue no longer races layers forward at decode speed). With no `Clock` it stays *read-paced* — exactly one inner frame per read — preserving the 1:1 passthrough a single-layer scaler (`OutputPresetVideoSource`) relies on, with no PTS-grid drift. The composite output PTS now carries the master time (`VideoCompositorSource.TryReadNextFrame` stamps `masterAlignmentTime` when provided) instead of a synthetic counter — this also upgrades the production cue path (`CueCompositionRuntime` already passes `masterPts`). Per-layer look-ahead buffers + converter are disposed on `RemoveLayer`/`Dispose` (`LayerHandle.Close`). Implemented the selection in the layer rather than the slot's 2-frame `MasterAligned` because consecutive submits collapse in the slot's single pending slot (can't hold a real PTS bracket). Regression tests `VideoCompositorCrossFadeTests.TimeDriven_SelectsFrameCoveringMasterTime_AndDecouplesFromReadRate` + `NoClock_IsReadPaced_OneInnerFramePerRead`. Verified no regression: Core 404, OpenGL 105, HaPlay 116 green.
